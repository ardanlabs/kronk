# Release Notes - v1.23.6

**Release Date:** April 24, 2026

## Overview

Version 1.23.6 introduces a major refactor of the configuration system to improve precision and differentiation between default and user-specified values. This change involves moving most configuration fields to pointers and providing a functional options API.

## Detailed Changes

### Breaking Changes

#### Configuration System Refactor (#517)

- **Author:** William Kennedy
- **Description:** A significant overhaul of how configuration is handled across the SDK and server.
- Most fields in `Config` and `DraftModelConfig` structs have been converted from direct values to pointers to allow distinguishing between zero values and unset values.
- Added getter methods (e.g., `Config.ContextWindow()`, `Config.NBatch()`, `Config.NSeqMax()`) to provide safe, non-pointer access to configuration values with default fallbacks.
- Introduced a new functional options pattern (`NewConfig` with `Option` functions) for a more idiomatic and robust configuration API.
- Updated all components (Server, SDK, examples, and CLI) to use the new configuration pattern.

### Improvements

#### Configuration Robustness

- Improved precision in model configuration, particularly for optional parameters like `RopeFreqBase`, `YarnExtFactor`, etc.
- Enhanced the ability to apply catalog configurations by differentiate between explicit user overrides and catalog defaults.

### Documentation

#### Manual and README Updates

- Updated documentation in `.manual/` and `README.md` to reflect the configuration changes.

### Upgrade Notes

#### Breaking Changes

The configuration API has changed. Fields in `Config` and `DraftModelConfig` structs are now pointers. Programmatic users of the SDK must now use the provided getter methods or the new `NewConfig` functional options API to interact with configuration.

#### Migration

- **For SDK Users:** Replace direct field access (e.g., `cfg.NBatch`) with the corresponding getter method (e.g., `cfg.NBatch()`).
- **For Configuration Construction:** Migrate from direct struct initialization to using `NewConfig` with `Option` functions for better type safety and clarity.
- **For YAML/JSON users:** The underlying structure for configuration files remains compatible, but the internal representation in Go has changed to support better differentiation of values.

#### Recommended Actions

- Audit all code that directly accesses `Config` or `DraftModelConfig` fields and update to use getters.
- Adopt the new functional options pattern when creating configuration objects in your applications.
